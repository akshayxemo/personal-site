The dream is simple: change a field in the database, and every layer that touches it fails to compile until you fix it. Here's how to get most of the way there without adopting a heavy framework.

## One schema, many uses

A single Zod schema can validate input, infer types, and document the shape. Don't define the same data three times.

```ts title="schema.ts"
import { z } from "zod"

export const createPost = z.object({
  title: z.string().min(1).max(120),
  body: z.string().min(1),
  tags: z.array(z.string()).default([]),
})

export type CreatePost = z.infer<typeof createPost>
```

## Validate at the edges, trust the middle

Parse untrusted input once at the boundary. After that, the type system carries the guarantee through your service layer.

<Callout>
  The boundary is anywhere data enters your system: request handlers, queue consumers, webhooks, and form actions. Parse there, and the rest of your code can assume valid data.
</Callout>

## Infer return types instead of annotating them

Let the compiler do the work. Explicit return types drift; inferred ones can't.

```ts title="query.ts"
export async function getPost(id: string) {
  const row = await db.posts.findOne({ id })
  return row // callers get the precise shape automatically
}
```

## Where to draw the line

Full end-to-end safety across a network boundary needs tooling (tRPC, generated clients). For everything inside a single process, inference plus Zod at the edges gets you 90% of the benefit at 10% of the cost.
