The hard part of a design system isn't the components. It's getting six teams with different deadlines to depend on code they don't own. Here's what moved the needle.

## Adoption is a product problem

Treat your internal teams like customers. They will only adopt the system if it's faster than rolling their own — so the migration path has to be genuinely easy.

```tsx title="before-and-after.tsx"
// Before: a bespoke button per team
<button className="btn btn-blue rounded-lg px-4 py-2">Save</button>

// After: one source of truth
<Button>Save</Button>
```

## Version with empathy

Breaking changes are a tax on every consumer. Batch them, document migrations, and provide codemods where you can.

<Callout type="warning">
  A major version bump that requires manual changes across six codebases can stall for months. Automate the migration, or don't ship the breaking change.
</Callout>

## Govern lightly, but govern

A design system with no rules drifts; one with too many rules gets bypassed. We landed on a small contribution guide, a single design review, and a public roadmap.

## Measure trust, not just usage

Usage tells you teams installed the package. Trust tells you they reach for it first. The real metric is how often teams build something custom *despite* the system existing — and driving that number down is the whole job.
