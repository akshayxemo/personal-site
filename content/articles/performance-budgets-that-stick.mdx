Every team has the same story: a fast launch, then a slow, invisible slide as features pile on. Performance budgets only work when they're enforced automatically and felt early.

## Measure what users feel

Optimize the Core Web Vitals, not synthetic vanity metrics. LCP, INP, and CLS map directly to perceived experience.

<Callout type="tip">
  Track field data (real users) alongside lab data. Lab tools catch regressions; field data tells you whether they actually hurt anyone.
</Callout>

## Put the budget in CI

A budget that lives in a doc is a suggestion. A budget in CI is a rule.

```js title="lighthouserc.js"
module.exports = {
  ci: {
    assert: {
      assertions: {
        "categories:performance": ["error", { minScore: 0.9 }],
        "largest-contentful-paint": ["error", { maxNumericValue: 2500 }],
      },
    },
  },
}
```

## Make regressions visible in the PR

The feedback has to arrive where the decision is made — in the pull request, not a weekly report nobody reads. A bundle-size comment on each PR changes behavior more than any dashboard.

## Ship the cheap wins first

- Defer non-critical JavaScript.
- Serve modern image formats at the right dimensions.
- Preconnect to critical origins.

None of these are glamorous, and together they routinely reclaim a second of load time.
